## Verified Facts

- The HyperFrames intro package has rendered landscape, bumper, and vertical
  cuts and uses a cream-first visual identity that matches the OneMem dashboard
  and landing-page palette.
- The Remotion demo package has current landscape and vertical
  dashboard-capture cuts, launch-mode landscape/square/vertical social cuts,
  and X-upload-safe 720p derivatives.
- The launch-mode cut is memory-first: it starts with runtime memory
  fragmentation, moves into a shared `MemoryNamespace`, then shows Seal,
  Walrus, Sui, runtime footage cards, and supported surfaces.
- The current proof/demo footage is dashboard trace-page capture generated from
  checked-in testnet trace artifacts. It is not final live proof of physical
  laptop switching, a real wallet transfer, live web research, or runtime hooks.
- The package already has strict live-proof scripts for manifest validation,
  Sui RPC object/transaction checks, shotlist generation, recording readiness,
  and live-proof Remotion rendering.
- The generated readiness JSON currently reports the final live-proof pass as
  not ready because the real manifest and four live clips are missing.
- Nautilus/TEE support exists as docs and a skeleton service only. The
  `services/nautilus-relayer` source prints that it is a skeleton/stretch
  component.
- No active video source should claim WASI, Nautilus, or TEE as shipped
  behavior without implementation and recording proof.

## Inferences

- The next honest production step is a final-video producer brief/spec, not an
  ungated render that pretends dashboard-capture footage is live proof.
- HyperFrames remains the right source for the emotional brand opener.
- Remotion remains the right source for final proof assembly because it already
  has footage modes, social profiles, render scripts, and strict live-proof
  gates.
- If "WASI code" means the core Sui/Walrus/Seal code path, the video should show
  the current substrate as Seal, Walrus/MemWal, and Sui. If it means literal
  WASI/TEE/Nautilus execution, the video should present it only as a future
  trust layer until implementation evidence exists.

## Unknowns And Questions

- Whether the final submitted video should include voiceover, or stay
  caption-first with music and sound design.
- Whether the final proof recordings will be captured as screen recordings,
  phone-camera physical-device footage, or a hybrid.
- Whether fresh testnet traces should be generated immediately before final
  recording to avoid stale visual evidence.

## Not Included

- No new video render was produced in this research pass.
- No live-proof manifest or live MP4 was created.
- No implementation claim is made for literal WASI, Nautilus, or TEE support.
