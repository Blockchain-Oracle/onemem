## User Flows

- New user arrives at landing, starts onboarding, signs in, creates account and
  namespace, installs runtime, runs an agent, verifies a trace.
- CLI user runs `onemem login`, opens browser pairing, selects namespace, generates
  scoped delegate key, returns to terminal.
- User inspects memory provenance from `/memories`, opens a memory drawer, follows
  its originating trace.
- User opens `/trace/[session_id]`, verifies chain integrity, inspects per-call
  content/proof metadata, optionally replays or shares.
- Public recipient opens `/verify/[session_id]` with no account and verifies chain
  integrity only.
- Owner shares a namespace capability and later revokes it.
- Runtime manager pauses/uninstalls a runtime or changes capture permissions.

## Revealed Product Requirements

- Verification has to be a first-class product moment, not a passive badge.
- The UI must distinguish what is proven from what is not proven.
- Runtime coverage tiers matter: native/full versus MCP/partial versus paused.
- Memory rows need provenance: app/runtime, agent, session, Walrus blob, Seal
  envelope, Sui tx, Merkle root, and access log.
- Cross-runtime sessions need a grouped view, not only individual TraceSession rows.
- CLI pairing needs a trustworthy browser flow with scoped delegate-key semantics.
- Settings needs delegate-key management and provider/runtime configuration.
- Share needs both owner and recipient modes, plus revoke state and activity history.

## Revealed Technical Requirements

- Dashboard needs data-layer boundaries for sessions, calls, memories, runtimes,
  capabilities, delegate keys, providers, and public verification.
- `/sessions` likely needs an aggregate/grouping model on top of raw TraceSession
  events.
- Runtime permissions imply a persisted local or hosted runtime config store.
- Pause/resume/uninstall imply per-runtime capture controls; current chain reads
  alone cannot perform those actions.
- Delegate-key management needs real key minting, TTL, labels, revocation, and a
  hosted auth boundary.
- Memory search/filter UI needs either MemWal-backed search or an honest degraded
  state when MemWal credentials are missing.
- Public verifier should never require login and should not attempt plaintext
  decryption.

## Data Model Candidates

- `RuntimeConnection`: id, runtime, coverage tier, version, status, lastSeenAt,
  sessionCount, memoryCount, permissions, pausedAt.
- `RuntimePermission`: autoCaptureMemory, autoRecall, traceAllCalls,
  notifyOnVerifyFailure.
- `MemoryRow`: id, text/decryptionState, class, tier, tags, app, agent, sessionId,
  walrusBlobId, sealEnvelopeHash, suiTxDigest, merkleRoot, verified, createdAt,
  accessLog.
- `UnifiedSession`: id/date, runtime lanes, subSessions, callCount, verifyStatus,
  windowStart, windowEnd.
- `DelegateKey`: id, label, ttl/expiresAt, namespaceId, createdAt, lastUsedAt,
  status.
- `CapabilityShare`: capabilityId, namespaceId, owner, recipient, kind, status,
  createdAt, revokedAt, activity.

## Auth, Permissions, And Security Implications

- Hosted login must separate Google/Enoki and wallet paths.
- CLI pairing must verify nonce and callback port and avoid sending fake credentials.
- Delegate keys need scoped TTL and namespace binding.
- Share/revoke must map to Sui capability semantics; revocation is not just a UI flag.
- Public verify should only prove Merkle chain integrity and must say plaintext,
  agent intent, and real-world correctness are not proven.
- Decrypt flows must remain client-side through Seal `/manual`; server plaintext is
  out of bounds.

## State And Edge Cases

Prototype states observed:

- Empty memory search.
- Loading/pending buttons.
- Paused runtime.
- Offline/idle/online runtime.
- Unverified, verified, and broken trace states.
- Locked/decrypting/decrypted content.
- Modal cancellation and escape-close behavior.
- Recipient versus owner share mode.
- CLI pairing success and manual fallback.

Additional states to plan:

- No Sui RPC or package unavailable.
- No MemWal credentials.
- Walrus blob unavailable.
- Seal decrypt denied because capability is missing or namespace inactive.
- Delegate key expired.
- Runtime installed but never seen on-chain.
- Unified session with one failed sub-session.

## Target-stack Translation

Do not copy the static prototype into production. Translate into:

- Next.js App Router routes in `packages/dashboard/app` and
  `apps/hosted-dashboard/app`.
- Server data helpers under `packages/dashboard/lib`.
- Client islands for modals, drawers, verify animation, replay, filters, tabs,
  and optimistic runtime settings.
- API routes for hosted/local mutations only where the product has a real storage
  or chain-backed action.
- Shared visual primitives in `packages/dashboard/components`, `apps/*/components`,
  and `packages/brand`.

## Spec Deltas

- Add explicit “public verify proves / does not prove” language to the verifier spec.
- Add unified session aggregation as a separate spec item rather than treating it as
  the same as raw `TraceSession`.
- Add runtime control semantics: pause, permissions, uninstall, and status source.
- Add delegate-key lifecycle semantics: mint, label, TTL, revoke, expired state.
- Add share recipient mode and revoke UX to capability-sharing specs.

## Story Deltas

- As a user, I can verify a trace without logging in and understand exactly what
  the verification proves.
- As a CLI user, I can pair my terminal with a scoped delegate key and namespace.
- As a runtime manager, I can pause a runtime without losing historical trace proof.
- As a memory user, I can inspect a memory’s provenance and jump to the originating
  trace.
- As an owner, I can share and revoke namespace access through Sui capabilities.
- As an auditor, I can verify all sub-sessions in a cross-runtime work session.

## Plan Deltas

- Treat the current UI as partially implemented, not greenfield.
- First plan pass should be an implementation-vs-prototype gap audit.
- Prioritize trust surfaces: public verify, trace drawer, decrypt content tab, and
  honest proof disclosure.
- Then implement operational surfaces: runtime controls, delegate keys, unified
  sessions, memory filters.
- Keep fake prototype actions out of production unless backed by real local config,
  hosted API, or chain operation.

## Quality Profile Deltas

- UI work needs browser verification for affected routes.
- Prototype-derived changes must include empty/loading/error/permission states.
- Files near 300 lines should be split before adding UI behavior.
- Public verifier changes need tests around success and failure states.
- Any decryption, delegate-key, capability, or runtime-control mutation requires
  security review and real-system testing.
